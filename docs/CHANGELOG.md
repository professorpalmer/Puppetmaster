# Changelog

## v0.9.0

**Truthful failure status, auto-fallback, live preflight, full-platform discovery, and real observability.** This release closes the gap between "looks healthy" and "actually verified before spending your money," and makes a degraded run re-route itself instead of silently lying.

- **Fixed a correctness defect: a preflight block / adapter billing rejection now FAILS the task.** `WorkerRuntime.run_once` previously discarded the `AgentRun` returned by `LocalWorker.run` and marked every dispatched task `COMPLETE` â€” so a preflight-blocked task (or an adapter that came back with a `billing_or_quota`/auth verification artifact) was recorded as a success. That made telemetry lie, made `await_job` report success over a doomed run, and made auto-fallback impossible. The runtime now honors the worker's `FAILED` verdict and converts any recoverable adapter failure (`RECOVERABLE_FAILURES`: billing/quota, preflight-blocked, missing key/CLI, auth) into a truthful `FAILED` status.
- **Auto-fallback on auth/billing failure.** After the main worker pass, the orchestrator re-routes any task that FAILED because *its adapter's account* was unfunded/unauthenticated to the cheapest sufficient model on a *different, verified-funded* adapter, then re-runs it (bounded: `_MAX_FALLBACK_ATTEMPTS=2`, `_MAX_FALLBACK_ROUNDS=3`). Each re-route writes a `router-fallback` ROUTING artifact and emits a `router.auto_fallback` event. Genuine errors (bad adapter, exceptions) still fail closed; a billing failure with no funded alternative surfaces loudly in the stitched **Alerts** section instead of nuking the whole job.
- **Live 1-token preflight probe** (`preflight.live_probe`, `preflight_check(..., live=True)`, `puppetmaster preflight --live`, per-task `payload.live_preflight`). Static detection can't see a funded-*looking* account whose balance is actually $0 â€” only a real call can. The probe does the smallest possible round-trip per adapter (Claude/Codex CLI, OpenAI `/v1/chat/completions`, Cursor = plan-catalog reachability) and classifies the result with the existing adapter classifiers. Off by default (adds a real call); fully dependency-injected for tests. **A probe only *blocks* on a definitive provider rejection (`billing_or_quota`/`auth`); a probe that can't reach a verdict (missing optional CLI, timeout, network blip) degrades to an "unverified" pass so `--live` never false-blocks a working account.** The OpenAI probe sends `max_completion_tokens` (GPT-5+ rejects the legacy `max_tokens`), with an automatic legacy fallback for older OpenAI-compatible endpoints.
- **Catalog discovery for OpenAI + Anthropic** (`puppetmaster/api_discovery.py`). Both expose `/v1/models`, so `python -m puppetmaster models discover --source openai|anthropic|all` now enumerates them too (additive merge â€” never drops hand-tuned entries; inherits capability/price from matching registry rows, seeds the rest). Cursor remains the default source.
- **Catalog staleness tracking.** `models discover --write` records per-source `{refreshed_at, count}` in a `models.discovery.json` sidecar; `doctor` gains a `catalog-freshness` check that nudges when a catalog is >30 days stale or never discovered.
- **Real observability** (`puppetmaster/telemetry.py`): OTel **metrics** (`puppetmaster.jobs` / `.tasks` counters + `cost_usd`), **live per-task spans** emitted by workers as they finish (`PUPPETMASTER_OTEL_LIVE`), and **cross-process trace-context propagation** â€” the orchestrator mints a W3C `traceparent`, exports it to worker subprocesses via `TRACEPARENT`, and anchors the assembled job span to it, so a swarm renders as one correlated trace in Jaeger/Datadog. All gated/zero-dep by default.
- **TypeScript blocking-await client** (`clients/typescript/`). `awaitJob(jobId, opts)` blocks until a job is terminal and returns its final state + stitched summary by driving the durable CLI (`python -m puppetmaster await --json`) â€” true blocking await for CI/scripts/backends, free of the MCP stdio constraint. Zero runtime deps; typechecks clean under `--strict`.
- **End-to-end validation (live, real adapters).** The marquee resilience flow was proven against real providers, not mocks: a `claude-code` worker hit a genuine `$0` Anthropic balance â†’ classified `billing_or_quota` â†’ the task was marked `FAILED` â†’ `router-fallback` re-routed it to `cursor/gpt-5.5` (plan-billed, `$0`) under the balanced policy â†’ the funded adapter **completed** the task (job `job_d82715bebc5d`; `created_by` audit trail: `worker-implement` â†’ `router-fallback` â†’ `worker-implement`). Concurrency was validated by `bench/mcp_stress.py` (**6/6** scenarios: 64 simultaneous senders, sustained traffic, large payloads, idleâ†’busyâ†’idle keepalive â€” 0 missing responses, 0 frame errors, server stayed alive). Benchmark receipts regenerated against this build: `router_savings` **35.1% cheaper** on a mixed workload (100% on routable tasks), `router_live_ab` **98.8% cheaper / 81.1% faster** with real OpenAI token receipts, `followup_cost` **0 adapter calls / 0 tokens** for 32 follow-up reads.
- **Bug fix (live probe false-block).** `preflight openai --live` blocked a fully-funded account because the probe sent the deprecated `max_tokens`, which GPT-5+ rejects with a 400 `unsupported_parameter` â€” misclassified as `probe_failed`. Fixed to `max_completion_tokens` (+ legacy fallback) and the block-only-on-rejection semantics above. Regression-tested.
- **Local dashboard** (`puppetmaster/dashboard.py`, `python -m puppetmaster dashboard [<job_id>]`). A zero-dependency live web board served from `http.server` â€” task graph, typed-artifact rollup, estimated cost, **auto-fallback reroutes**, and the same Alerts the stitcher surfaces. The page is fully inlined (no Flask/React/CDN; works offline) and polls every ~1.5 s, so it's live while a swarm runs and instant for a finished one. OTel remains the export path for Jaeger/Datadog; this is the no-collector local view. Data layer (`build_job_snapshot`) is a pure function; an HTTP smoke test drives the real server. **Hardened via dogfooding:** a Cursor swarm reviewing this very module (which itself survived a live `claude-code` `$0` failure â†’ `router-fallback` â†’ `openai/gpt-5.5` completion) flagged three real issues, now fixed â€” `/api/job` rejects non-job-id requests before the store path join (no `..`/absolute traversal), the catch-all handler returns a generic 500 instead of leaking `str(exc)` (paths/backend details), and the cost rollup tolerates a non-numeric `estimated_cost_usd` instead of 500-ing.
- **Cross-platform CI.** The GitHub Actions matrix now runs the full suite + CLI/demo smokes on **Linux, macOS, and Windows** (Python 3.9 + 3.12) with `shell: bash` so the heredoc steps run identically everywhere. Fixed a Windows-only runtime breaker: `mcp_registry` hard-kill used `signal.SIGKILL` (absent on Windows â†’ `AttributeError`); now falls back to `SIGTERM` (which `os.kill` maps to `TerminateProcess` on Windows).
- **Demo.** `scripts/demo.sh` â€” a 60-second, no-API-key tour (cost routing â†’ 6-role swarm fan-out â†’ stitched summary â†’ `$0` follow-up reads â†’ dashboard pointer) that runs on a clean machine; `scripts/demo.tape` regenerates `docs/demo.gif` via VHS (GIF-as-code), and the rendered `docs/demo.gif` (~1.1 MB, 24 fps) is committed so the README shows it without a local VHS install.

## v0.8.0

**Platform-contained billing + dynamic Cursor catalog discovery.** Puppetmaster now treats the subscription you already pay for (Cursor plan, Claude Max, ChatGPT/Codex) as the default place to spend, and can discover what your Cursor plan actually exposes instead of relying on a hand-maintained registry. Also fixes a **critical packaging bug** that broke the Cursor adapter for every `pip install` user.

> **Packaging fix (affects all prior PyPI releases):** `puppetmaster/cursor_sdk_runner.mjs` â€” the Node runner the Cursor SDK adapter resolves via `Path(__file__).with_name(...)` â€” was **never included in the wheel/sdist** (no `package-data` rule), so `pip install puppetmaster-ai` shipped a Cursor adapter that couldn't launch. Added `[tool.setuptools.package-data] puppetmaster = ["*.mjs"]`; verified the runner is now present in both wheel and sdist and resolvable after a clean-venv install. **Anyone using the Cursor adapter from a pip install must upgrade to 0.8.0.**

- **First-class `ModelSpec.billing`** (`plan` | `api` | `unknown`, default `unknown`). Plan = covered by a subscription (no marginal API spend); api = out-of-pocket per-token; unknown = resolve at runtime. Serializes only when non-default; round-trips through `load_registry`/`save_registry`. Starter registry tags Cursor entries `plan` and OpenAI entries `api`. New `is_plan_billed` property.
- **Billing-aware router.** `TaskSignals` gains `prefer_plan_billed` (default **True**) and `allow_api_billing` (default **True**). At sufficient capability the router prefers a plan-billed model over an out-of-pocket one (tie-break across `cheap`/`balanced`/`quality`/`escalating`); `allow_api_billing=False` is a hard plan-only gate that rejects api-billed models with an explicit reason and raises `NoEligibleModelError` if nothing plan-billed qualifies. The picked model's `billing` is now stamped into the `ROUTING` artifact for audit. Overridable per task via `payload.prefer_plan_billed` / `payload.allow_api_billing`.
- **Cursor catalog discovery** (`puppetmaster/cursor_discovery.py` + `cursor_sdk_runner.mjs` `mode: "list-models"`). `python -m puppetmaster models discover` enumerates the authenticated Cursor plan's models via `Cursor.models.list()` and reconciles them into the registry as **plan-billed** entries (`--write` to persist, dry-run by default, `--json` for machines). Cross-adapter **capability inheritance**: a model the plan exposes (e.g. `claude-opus-4-8`) inherits its native capability/context (cap 99) by model-name match, so a plan-billed frontier ranks like the frontier it is. Stale cursor entries no longer in the catalog are dropped; non-cursor entries pass through untouched. *Live result on the developer's plan: 28 plan-billed models discovered, and a hard audit re-routed from `claude-code/opus-4-8` (api, ~$0.13/call) to `cursor/claude-opus-4-8` (plan, $0).*
- **Platform billing/auth detection** (`puppetmaster/platform_billing.py`). Pure, dependency-injected probes for each adapter: Cursor (keyed â†’ plan), Claude Code (`ANTHROPIC_API_KEY` â†’ api, else `~/.claude` OAuth â†’ plan), Codex (`codex login status`: API key â†’ api, ChatGPT â†’ plan), OpenAI (api). Returns `BillingStatus{billing, healthy, detail, evidence}`. `doctor` now prints a `billing:<adapter>` line per platform.
- **Universal preflight** (`puppetmaster/preflight.py` + `python -m puppetmaster preflight <adapter> [--model] [--no-api-billing]`). Verifies an adapter can actually run *before* dispatch â€” credentials present, plan-vs-api gate, and (Cursor) the routed model is in the live catalog. Catalog-fetch failures degrade to a pass with a note (never block a run because discovery was offline). This is the safety net for the "returned in 391ms with billing failure and zero edits" degraded-run class.
- **Loud failure surfacing.** The stitched summary now leads with an **`## Alerts (action required)`** section that hoists hard failures (`billing_or_quota`, `missing_api_key`, `missing_cli`, `model_unavailable`, `permission_denied`, â€¦) to the top with remediation, instead of burying them in a low-confidence verification line.
- **`await_job`.** New `puppetmaster_await_job` MCP tool (bounded long-poll: blocks up to `timeout_seconds` for a terminal state, returns `timed_out` so you can re-call) and `python -m puppetmaster await <job_id>` (true blocking await for the CLI/SDK path, `--timeout-seconds 0` blocks until done).
- **Pre-dispatch preflight gate.** `LocalWorker` now runs the fast (no-network) auth/billing check before dispatching to a paid adapter (cursor/claude-code/codex/openai). An unauthenticated/missing adapter â€” or an api-billed one under a plan-only task â€” yields a `blocked` verification artifact (`failure=preflight_blocked`) and a FAILED run instead of a wasted dispatch; opt out with `payload.skip_preflight`. Pairs with billing-aware routing (auto_route jobs already steer hard work to the plan-billed Cursor frontier, so they avoid a depleted API account entirely). Surfaced in the stitcher Alerts section. *Live token probes for the funded-but-zero-balance case remain a follow-up.*
- **Optional OpenTelemetry tracing** (`puppetmaster/telemetry.py`, `pip install 'puppetmaster-ai[otel]'`). Zero-cost and zero-dependency unless enabled via the standard env vars (`OTEL_EXPORTER_OTLP_ENDPOINT`, or `OTEL_TRACES_EXPORTER=console`, with a `PUPPETMASTER_OTEL_ENABLED` override). On completion each job emits a trace â€” a `puppetmaster.job` session span with `puppetmaster.task` children carrying role, adapter, `gen_ai.request.model`, status, and `puppetmaster.cost_usd`/`gitagent.cost_usd` â€” assembled from durable records (faithful per-worker tree without threading span context across the subprocess boundary) and exported via OTLP or printed to the console. Trace assembly (`build_job_trace`) is a pure, SDK-free function; a telemetry failure never breaks a swarm. Verified end-to-end with the console exporter.
- **Bug fix:** `_server_version()` looked up the dist as `puppetmaster`, but it's published as `puppetmaster-ai` â€” pip users got `None` in MCP `serverInfo.version`. Now tries `puppetmaster-ai` first, falls back to the legacy name.
- **Tests:** +27 tests (`214 â†’ 241`) across billing field round-trip, billing-aware routing (plan tie-break, api gate), platform detection (all auth branches), Cursor discovery (catalog fetch/failure modes, overlay + cross-adapter inheritance, stale-drop merge), preflight (ready/blocked/catalog-unavailable), stitcher alerts, and `await_job_state` (terminal + timeout). Full suite green; clean-venv install of the built wheel verified end-to-end.

Idempotent: if `codex mcp get puppetmaster` already reports a matching command + args, returns `unchanged` without touching `~/.codex/config.toml`. Prints the Codex sandbox guidance (interactive TUI approval flow vs `--dangerously-bypass-approvals-and-sandbox` for `codex exec`) as part of "next steps" output.

## v0.7.0

- **Codex is no longer a stub â€” full Codex CLI integration shipped.** The `codex` slot has been a placeholder `UnconfiguredProviderAdapter("codex", "Codex")` since `puppetmaster/adapters.py` was first written. v0.7.0 promotes it to a real adapter, sibling to `CursorAdapter`, `ClaudeCodeAdapter`, and `OpenAIAdapter`. Puppetmaster now ships **four production adapters** instead of three.
- **`CodexAdapter` shells out to the official OpenAI Codex CLI (`codex exec --json`).** Required: `npm install -g @openai/codex`, then `printenv OPENAI_API_KEY | codex login --with-api-key` (or `codex login` for ChatGPT-account auth). The adapter is modeled on `ClaudeCodeAdapter` for subprocess + git-snapshot + sidecar-spool semantics, with one major addition: it parses Codex's structured JSONL event stream and captures **real billing-grade token telemetry** into the verification artifact payload â€” `input_tokens`, `output_tokens`, `cached_input_tokens`, `reasoning_output_tokens`, and `thread_id` per turn. This makes `codex` strictly more telemetry-rich than any other adapter Puppetmaster ships; `claude-code` doesn't surface usage at all, and `openai` only surfaces the basic prompt/completion split.
- **Default execution mode is non-interactive and safe.** `approval_policy="never"` (no automation hangs on approval prompts), `--sandbox workspace-write` (can edit files in `cwd`, not outside), `--ephemeral` (no persistent Codex sessions), `--skip-git-repo-check` (works in non-git directories). Callers can downgrade to `payload.sandbox="read-only"` for review/plan roles, or opt in to `payload.dangerously_bypass_approvals_and_sandbox=true` when the surrounding environment is already externally sandboxed.
- **Two new starter registry tiers.** `codex/gpt-5-5` (capability_score 97, $5/$30 per MTok, 1M context, tags include `frontier` + `agent-loop`) and `codex/gpt-5-4-mini` (capability_score 72, $0.75/$4.50, 400K context, tags include `balanced` + `agent-loop`). Capability scores are set 1 point higher than the matching `openai/*` tiers to encode the agent-loop advantage on multi-file refactors and audits. - **Cross-workspace job lookup for read-only CLI commands.** `puppetmaster show`, `artifacts`, `diff`, `feed`, `logs`, `events`, `status`, `memory`, and `open` now auto-pivot to the project state dir that owns a given `job_id` if it isn't present in the current workspace's state dir. No more `PUPPETMASTER_STATE_DIR=~/Library/.../ff-data-engineering-cfbfad67d9fc python -m puppetmaster show ...` â€” just paste the job id from any cwd. Explicit `--state-dir` or `$PUPPETMASTER_STATE_DIR` overrides still take precedence (write-side commands always use the caller's workspace, never pivot).
- **New `puppetmaster projects` command + `puppetmaster jobs --all-projects` flag.** `projects` lists every Puppetmaster project state dir on this machine with job counts and last activity. `jobs --all-projects` flattens jobs from every project into one stream with a project column. Together they make "which workspace owns this job?" a one-liner.
- **Orphan self-termination via stdin staleness watcher.** Cursor's MCP "lease" lifecycle periodically re-creates the logical client without killing the previous Python server, so before this release Puppetmaster could accumulate one MCP server per lease cycle â€” each still holding SQLite handles and competing for the CodeGraph indexer lock. The new `_InputStalenessWatcher` measures inbound JSON-RPC traffic directly (not just process liveness like the heartbeat thread): if no stdin message arrives for 10 minutes **and** there are zero in-flight tool calls, the watcher closes stdin so the main loop sees EOF and exits through the normal `finally` block (deregister, stop heartbeat, shut down executor). No `os._exit`, no leaked tracking files. Tunable via `PUPPETMASTER_MCP_INPUT_STALE_SECONDS`, `PUPPETMASTER_MCP_INPUT_STALE_CHECK_SECONDS`, and `PUPPETMASTER_MCP_INPUT_STALE_DISABLED`.
- **Explicit Cursor-Node invocation for CodeGraph.** New `resolve_codegraph_invocation()` in `puppetmaster.codegraph` returns `[cursor_node, codegraph.js]` whenever Cursor's bundled Node and the global `@colbymchenry/codegraph` install are both discoverable. Every subprocess that previously invoked the bare `codegraph` shim â€” `codegraph_context`, `codegraph_status_line`, `run_codegraph_cli`, and the background `codegraph_index_runner` â€” now invokes CodeGraph under Cursor's Node first, with a transparent fallback to the PATH shim when Cursor isn't installed (Linux, headless CI). This closes the 18-hour-runaway-indexer class of bug, where a stray shell invocation under Homebrew Node would fall back to WASM SQLite, lock the DB, and burn 99% CPU for hours. `PUPPETMASTER_CODEGRAPH_NODE` / `PUPPETMASTER_CODEGRAPH_JS` env vars are an escape hatch for non-standard installs.
- **Doctor distinguishes shell-Node WASM from Cursor-Node WASM.** `puppetmaster doctor`'s `codegraph` check used to warn whenever the shim on PATH reported WASM, even after `repair-codegraph` had successfully rebuilt better-sqlite3 for Cursor's Node â€” leading to misleading "broken" diagnoses when MCP was actually healthy. The check now runs against the same invocation Puppetmaster uses at runtime (`resolve_codegraph_invocation()`) and reports `ok (verified under Cursor's bundled Node)` when the runtime that matters is healthy.
- New tests covering invocation-resolution under Cursor Node, environment overrides, the fallback path, staleness-watcher trigger/hold-off/reset, env-disable, dispatcher counter, doctor's Cursor-Node disambiguation, SDK detection from the package install dir, and cross-workspace job lookup. Total suite at 125 tests, all passing.

## v0.5.3-beta.1

- **Tool-call keepalive notifications.** Every long-running MCP `tools/call` now spawns a per-call `_ToolCallKeepalive` daemon thread that emits MCP-spec `notifications/message` frames every 10 seconds (after a 5-second grace period) for as long as the handler runs. Short calls pay zero protocol cost; the goal is to keep bytes flowing on the stdio pipe so Cursor's MCP client stops treating slow tool calls as a dead transport. This is the prevention complement to v0.5.2's cleanup â€” together they cover both "stop the drop from happening" and "recover cleanly when it does."
- All keepalive frames are unidentified JSON-RPC notifications (no `id`) and carry a structured `data.kind = tool_call_progress` payload with the tool name, request id, and elapsed time so logs and progress UIs can render them.
- Tunable via `PUPPETMASTER_MCP_KEEPALIVE_AFTER_SECONDS`, `PUPPETMASTER_MCP_KEEPALIVE_INTERVAL_SECONDS`, and `PUPPETMASTER_MCP_KEEPALIVE_DISABLED`. Default behaviour is on and conservative.
- Broken-pipe detection: if the emitter sees a dead stdio (`BrokenPipeError`/`OSError`), the keepalive thread shuts down cleanly instead of spinning. Post-response races are also closed via a stop-check inside the `_STDOUT_LOCK`.
- README troubleshooting section now distinguishes the prevention layer (this release) from the recovery layer (v0.5.2 mcp registry + cleanup tools).

## v0.5.2-beta.1

- **MCP server process registry + orphan cleanup.** Every Puppetmaster MCP server now writes a tracking file (PID, workspace, started_at, last_heartbeat) at startup, bumps its heartbeat from a daemon thread, and removes its file on a clean exit. New servers prune dead tracking files on startup so orphans from prior `Tool execution error. Not connected` cycles get reaped automatically â€” no more `pkill -f puppetmaster.mcp_server` from a shell.
- New `puppetmaster/mcp_registry.py` module with `register()`, `heartbeat()`, `deregister()`, `list_entries()`, `prune_dead()`, `kill_stale()`, and a `HeartbeatThread`. Per-user storage lives under `~/Library/Caches/puppetmaster/mcp-servers/` on macOS and `$XDG_CACHE_HOME/puppetmaster/mcp-servers/` on Linux; overridable via `PUPPETMASTER_MCP_REGISTRY_DIR`.
- New CLI surface: `python -m puppetmaster mcp list` (with `--json`) and `python -m puppetmaster mcp cleanup` (`--kill-stale`, `--stale-after-seconds`, `--json`). Lists every tracked server with alive/stale/dead flags and reclaims dead entries; `--kill-stale` SIGTERMs (and SIGKILLs after a grace period) servers whose heartbeat is older than the threshold. The current PID is never signalled.

- `puppetmaster doctor` now includes an `mcp-servers` check that warns when dead tracking files or stale-but-alive servers are present, and points at the new cleanup command. This is the smoking gun behind a stale `Tool execution error. Not connected` symptom.
- README: new "`Tool execution error. Not connected` from Cursor" Troubleshooting section explaining the failure mode (Cursor closes the stdio transport while the swarm keeps running), the diagnostic command, the cleanup command, and the durable-state implication that orphan reconnects are recoverable.
